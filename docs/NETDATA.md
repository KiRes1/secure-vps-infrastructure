\## 1. Установка



```bash

apt update

apt install netdata -y

```



Установка занимает 1-2 минуты. После установки Netdata автоматически запускается.



\---



\## 2. Настройка безопасности



Netdata по умолчанию слушает порт `19999` на всех интерфейсах. Это небезопасно.



\### Привязать к localhost:



Проверить текущую конфигурацию:



```bash

grep "bind" /etc/netdata/netdata.conf

```



Должно быть:



```ini

bind socket to IP = 127.0.0.1

```



Если нет — добавить в `/etc/netdata/netdata.conf`:



```bash

nano /etc/netdata/netdata.conf

```



В секции `\[web]`:



```ini

\[web]

&#x20;   bind to = 127.0.0.1

```



\### Перезапустить:



```bash

systemctl restart netdata

```



\### Проверить:



```bash

ss -lntp | grep 19999

```



Правильный вывод:



```

LISTEN 0 4096 127.0.0.1:19999 ...

```



Неправильный (доступен из интернета):



```

LISTEN 0 4096 0.0.0.0:19999 ...

```



\### Закрыть порт в файрволе:



```bash

ufw deny 19999/tcp

```



\---



\## 3. Доступ через SSH-туннель



\### На локальном компьютере (Windows PowerShell):



```powershell

ssh -N -T -L 19999:127.0.0.1:19999 root@<IP> -i <путь\_к\_ключу>

```



Окно не закрывать.



\### Открыть в браузере:



```

http://127.0.0.1:19999

```



\### Для удобства добавить в SSH Config:



```

Host netdata-ru

&#x20;   HostName <RU\_IP>

&#x20;   User root

&#x20;   IdentityFile <путь\_к\_ключу>

&#x20;   LocalForward 19999 127.0.0.1:19999

&#x20;   SessionType none

&#x20;   ExitOnForwardFailure yes

```



Запуск:



```powershell

ssh -N netdata-ru

```

\## 4. Что мониторим



| Метрика | Что показывает | Признак проблемы |

|---------|---------------|-----------------|

| \*\*CPU\*\* | Загрузка процессора | Резкий скачок → DDoS или атака |

| \*\*RAM\*\* | Использование памяти | Постоянный рост → утечка памяти |

| \*\*Network\*\* | Входящий/исходящий трафик | Падение до нуля → обрыв сети |

| \*\*Disk I/O\*\* | Нагрузка на диск | Высокий I/O → проблема с БД |

| \*\*Connections\*\* | Количество подключений | Резкое падение → клиенты отвалились |

| \*\*Uptime\*\* | Время работы сервера | Сброс → перезагрузка сервера |

\## 5. Диагностика падения VPN



\### Если VPN упал:



1\. Запустить туннель к Netdata

2\. Открыть `http://127.0.0.1:19999`

3\. Выбрать временной промежуток (когда упал VPN)

4\. Посмотреть графики:



| Симптом | Вероятная причина |

|---------|------------------|

| Скачок CPU до 100% | DDoS-атака |

| Сеть упала до 0 | Проблема у провайдера |

| Резкий рост трафика | Кто-то качает торренты |

| Перезагрузка сервера (Uptime сброшен) | Краш системы или OOM |

| Все графики в норме | Проблема на стороне клиентов или блокировка |



\### Дополнительная диагностика:



```bash

\# Логи Xray за проблемный период

journalctl -u x-ui --since "2026-07-22 12:00" --until "2026-07-22 14:00" --no-pager



\# Статус сервисов

systemctl status x-ui nginx --no-pager



\# Доступность NL-сервера

ping -c 3 <NL\_IP>

```

\## 6. Проверка работы



```bash

\# Статус сервиса

systemctl status netdata --no-pager



\# Порт слушает

ss -lntp | grep 19999



\# Логи (нет ошибок)

journalctl -u netdata -n 20 --no-pager

```



\---



\## Примечание



Netdata НЕ отслеживает:

\- Статус отдельных клиентов VPN

\- UUID или подписки

\- Конфигурацию Xray



Для этого используются:

\- Панель 3x-ui (клиенты)

\- `systemctl status x-ui` (сервис)

\- `ss -lntp | grep 443` (порт)

\- `journalctl -u x-ui` (логи)

## 1. Общие принципы

Подписка — это URL, который клиентское приложение периодически запрашивает для получения актуального конфига.

**Правильная схема:**

```
Клиент → HTTPS → Nginx (Reverse Proxy) → Subscription Service (внутренний)
```

**Неправильные схемы:**

- ❌ Подписка по HTTP (конфиг передаётся открытым текстом)
- ❌ Стандартный путь `/sub/` (легко обнаружить)
- ❌ Прямой доступ к порту подписки из интернета
- ❌ Ссылки с `localhost` или `127.0.0.1`

## 2. Настройка в панели 3x-ui

### Путь: Настройки панели → Подписка

| Параметр | Рекомендация | Пример |
|----------|-------------|--------|
| **Прослушивание IP** | `127.0.0.1` (если через Reverse Proxy) или `0.0.0.0` | `127.0.0.1` |
| **Домен прослушивания** | Внешний IP сервера или домен | `<IP_сервера>` |
| **Порт подписки** | Нестандартный порт | `2097` |
| **URI-путь** | Уникальный, не `/sub/` | `/mysub_xk39fh28dh37dh/` |
| **URI обратного прокси** | Полный URL (если Nginx на другом порту/домене) | `https://<IP>:2096/mysub_xk39fh28dh37dh/` |

### ⚠️ Важно:

- **Не использовать** стандартный путь `/sub/` — он широко известен и сканируется ботами
- **Всегда использовать HTTPS** для публичного доступа (через Nginx)
- **Домен прослушивания** должен быть внешним IP или доменом, иначе ссылки будут генерироваться с `localhost`

## 3. Формат ссылок

### Ссылка подписки (для клиентских приложений):

```
https://<IP>:2096/<уникальный_путь>/<subId_клиента>
```

**Пример:**
```
https://<IP>:2096/mysub_xk39fh28dh37dh/h0kga58ropwa9v7l
```

Клиент вставляет эту ссылку в поле «Subscription URL» или «URL подписки».

### Прямая ссылка (единоразовый импорт):

```
vless://<UUID>@<IP>:443?encryption=none&flow=xtls-rprx-vision&fp=firefox&pbk=<public_key>&security=reality&sid=<short_id>&sni=<sni>&type=tcp#<имя>
```

Клиент вставляет как «Импорт конфига» или сканирует QR-код.

## 4. Исправление localhost в ссылках

### Проблема:

Ссылки генерируются с `localhost` или `127.0.0.1`:

```
http://127.0.0.1:2097/sub/h0kga58ropwa9v7l  ← не работает на телефоне
vless://...@localhost:443?...                ← не работает на телефоне
```

### Причина:

Поле **Домен прослушивания** пустое.

### Решение:

1. Зайти в панель → **Настройки панели** → **Подписка**
2. Заполнить поле **Домен прослушивания**:
   - Внешний IP сервера (например, `185.182.65.78`)
   - Или домен (например, `sub.example.com`)
3. Сохранить
4. Перезапустить x-ui:

```bash
systemctl restart x-ui
```

5. Зайти в клиента → нажать **Копировать ссылку** заново

После исправления ссылки будут содержать правильный IP/домен.

## 5. Безопасность

### Что защищает подписку:

| Уровень | Защита |
|---------|--------|
| **HTTPS** | Шифрование трафика (Nginx + Let's Encrypt) |
| **Уникальный URI** | Затрудняет обнаружение ботами |
| **SubId** | Длинный уникальный идентификатор клиента |
| **Reverse Proxy** | Прямой порт подписки закрыт, доступ только через Nginx |

### Рекомендации:

- ✅ Использовать уникальный URI-путь (не `/sub/`)
- ✅ Использовать HTTPS (через Nginx)
- ✅ Закрыть прямой порт подписки в файрволе
- ✅ Регулярно проверять срок сертификата
- ❌ Не передавать ссылки подписки в открытых каналах
- ❌ Не использовать стандартные порты

## 6. Диагностика проблем

### Подписка возвращает 404

```bash
# Проверить, что сервис слушает
ss -lntp | grep 2097

# Проверить, что путь правильный
curl -I http://127.0.0.1:2097/<URI_путь>/<subId>
```

### Подписка возвращает 502

```bash
# Проверить Nginx
systemctl status nginx

# Проверить, что x-ui запущен
systemctl status x-ui
```

### Подписка не открывается извне

```bash
# Проверить, что Nginx слушает порт 2096
ss -lntp | grep 2096

# Проверить файрвол
ufw status | grep 2096

# Проверить сертификат
openssl s_client -connect <IP>:2096 -servername <IP> </dev/null 2>/dev/null | openssl x509 -noout -dates
```

### Клиент получает n/a пинг

1. Проверить, что в конфиге указан внешний IP, а не `localhost`
2. Проверить, что на телефоне установлен «чистый» DNS (например, `1.1.1.1`)
3. Проверить, что порт 443 открыт и слушает `0.0.0.0`

### Ссылка всё ещё содержит localhost после исправления

1. Сохранили ли вы настройки?
2. Перезапустили ли x-ui? `systemctl restart x-ui`
3. Скопировали ли ссылку **заново**? (старая ссылка не обновится автоматически)

## Памятка владельцу

При создании нового клиента:

1. Клиент создаётся в панели RU-сервера
2. Ему назначается уникальный SubId
3. Формируется ссылка: `https://<IP>:2096/<URI>/<subId>`
4. Nginx автоматически проксирует запрос
5. Конфигурацию Nginx редактировать не нужно
6. На NL-сервере нового клиента создавать не нужно

---

*Все IP, порты и идентификаторы заменены на плейсхолдеры.*
